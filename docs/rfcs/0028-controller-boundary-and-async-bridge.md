# RFC-0028: Controller Boundary & Async Bridge — making `#[byard_controller]` real

- **Status:** Draft
- **Author(s):** Briany4717
- **Created:** 2026-07-17
- **Last updated:** 2026-07-17
- **Depends on:** RFC-0001 (§5.1 thread layout / Tokio I/O pool, §"Rust controller", the two-layer split, INV-1 crate dependency direction), RFC-0002 (`inject`, `Env::provide`, `Value`), RFC-0003 (events / `Action`, reference-freedom, E1 write-back), RFC-0004 (reactive tick, `apply_io_results` timing), RFC-0019 (callback props / `Fn` — the result-handler arms), RFC-0027 (`Value::Record`, the data shape controllers return).
- **Extends:** RFC-0001 §5.1 (turns the "Tokio pool executes async I/O from controllers" sentence into a wired path), the `byard-macro` crate (the `#[byard_controller]` macro gains a dispatch shim), `relay.rs` (the logic-thread loop drains I/O results into the interpreter).
- **Enables:** Any `.byd` view calling into Rust for networking, disk, OS, or long compute and receiving the result reactively. Hard prerequisite for RFC-0029 (async I/O capabilities) to be reachable from `byld`.
- **Amended by:** [erratum `0001-erratum-controller-boundary.md`](0001-erratum-controller-boundary.md) (corrects the pre-existing `#[byard_controller]` doc-comment).

---

## Summary

RFC-0001 §5.1 promises a Tokio pool that "executes async I/O from Rust
controllers \[and] sends results back to the logic thread via
`tokio::sync::mpsc`." Today that path is **not wired end to end**: the
`#[byard_controller]` macro only emits field-type metadata
(`byard-macro/src/lib.rs`), `inject` resolves ambient *values* but cannot
resolve a *controller handle*, `byld` has no syntax to *call* a controller, and
the logic thread's tick loop (`relay.rs::spawn_logic_from_view`) never drains the
I/O result channel — the channel's only real consumer is image decode on the
**render** thread (`engine.rs:610`). This RFC specifies the whole boundary: a
`Controller` trait and a **Send-only `HostValue` wire type** in `byard-core`
(keeping INV-1 intact), a real dispatch shim generated by
`#[byard_controller]`, a controller **registry** wired through `inject`, a
`byld` **async-call expression** with typed success/error result arms, and the
missing plumbing that drains `IoResult` at the top of each logic tick into
`Interpreter::apply_io_results`, which writes the bound `var` and marks it dirty.
Nothing crosses a thread boundary except `Send` data (INV-2); all blocking work
stays on the Tokio pool (INV-12).

```byld
View Weather() {
    inject WeatherApi as api          // resolves a controller handle
    var report: Record = { state: "idle" }

    Column #[gap: 12, p: 16] {
        Button("Load Tokyo") => api.forecast("Tokyo")
            ok  report => { report = { ..report, state: "ok", data: report } }
            err e      => { report = { state: "error", message: e.message } }

        when report.state == "ok" {
            Text("{report.data.tempC}°C")
        }
    }
}
```

```rust
// weather.rs — the controller half of the app
#[byard_controller]
pub struct WeatherApi { base_url: String }

impl WeatherApi {
    pub async fn forecast(&self, city: Str) -> Result<Forecast, ApiError> { /* RFC-0029 */ }
}
```

---

## Motivation

The [gap analysis](../../support/GAP_ANALYSIS_real_apps.md) names this the
decisive blocker: *"there is no path for the UI to ask Rust for data and receive
the answer."* Every non-trivial app — weather, YouTube-like, even a persistent
todo list — needs the `.rs` half of the two-file model to actually run. The
scaffolding exists but is disconnected:

- **The macro is inert.** `#[byard_controller]` emits `BYARD_FIELDS` +
  `byard_field_type()` and nothing else (`byard-macro/src/lib.rs:65-89`). Its own
  doc-comment claims it "exposes async methods, provides ambient values to
  `inject`, and delivers results to the logic thread" — none of which the code
  does. (Corrected by the erratum.)
- **`inject` cannot reach Rust.** `Env::provide`/`resolve_inject` store and fetch
  a `Value` within the `byld` tree (`interp/env.rs:159-193`); there is no
  controller handle and no way to invoke a method.
- **No call syntax.** An `=> action` can only mutate signals synchronously on the
  logic thread (`Action = Box<dyn FnMut(&mut ReactiveCtx, Option<&Value>)>`).
- **The result channel is orphaned on the logic side.** `Relay` exposes
  `io_result_sender()`/`try_recv_io_result()` and `Interpreter::apply_io_results`
  exists (`eval.rs:995`), but the logic loop never connects them; only the render
  thread drains the channel, and only for `DecodedImage`.

This RFC turns RFC-0001's promise into a tested, reference-free, `!Send`-safe
mechanism.

## Guide-level explanation

### 1. A controller is a Rust struct with async methods

The developer writes an ordinary struct, annotates it, and gives it async
methods returning `Result<T, E>`. `#[byard_controller]` makes it dispatchable
from `byld` **and** keeps the field metadata it already produced for LSP
autocompletion. The struct is emitted unchanged (INV-1: the macro adds an
`impl`, never a back-dependency).

### 2. The app registers controllers, then runs the view

An app is a crate whose `main.rs` builds controllers, hands them to the engine as
**providers** (the `inject` provider side), and points the engine at the `.byd`
entry. `byard new` scaffolds exactly this (today it scaffolds only `main.byd` —
fixed here):

```rust
fn main() -> Result<(), ByardError> {
    let api = WeatherApi { base_url: "https://api.example.com".into() };
    byard::App::new("main.byd")
        .provide(api)          // ambient, resolved by `inject WeatherApi as …`
        .run()
}
```

### 3. `inject` resolves the handle; a call spawns async work

`inject WeatherApi as api` resolves a **controller handle** (not a value copy).
`api.forecast("Tokyo")` inside an event action does **not** block: it packages
the arguments as `HostValue`s, spawns the real async method on the Tokio pool,
and registers a **continuation**. The action returns immediately; the view keeps
rendering the current frame.

### 4. The result comes back on the logic thread, into a `var`

When the async method completes, its `Result` is delivered — as `Send`
`HostValue` — to the logic thread, which runs the matching **result arm** on the
next tick:

- `ok  name => { … }` runs on success, binding the returned value as `name`.
- `err e => { … }` runs on failure, binding the error record as `e`.

The arm body is an ordinary action: it writes `var`s, which mark dirty, which
pull next tick — the standard reactive update path. There is no polling, no
`await` in `byld`, and no shared mutable state.

### 5. Ambient values still work

`provide(value)` for a plain `Str`/`Int`/`Record` remains the existing `inject`
ambient-value mechanism (RFC-0002). Controllers are simply providers whose
ambient value is a handle. Theme injection (RFC-0022) is unchanged.

## Reference-level explanation

### 1. The `HostValue` wire type (in `byard-core`, INV-1/INV-2)

`Value` (with `Signal`/`Memo`/`Fn`) lives in `byard-compiler` and is `!Send`. The
boundary therefore uses a neutral, `Send`, serialization-free data type defined
in `byard-core`, so the trait that both the app crate and the interpreter speak
does not drag `byard-compiler` into `byard-core` (INV-1):

```rust
// byard-core::bridge
pub enum HostValue {
    Unit,
    Bool(bool),
    Int(i64),
    Float(f64),
    Str(String),
    List(Vec<HostValue>),
    Record(Vec<(String, HostValue)>),
}
// Compile-time: HostValue: Send + 'static  (static assertion, INV-2)
```

The interpreter provides total, lossless conversions `Value ⇄ HostValue` for the
data subset (signals/memos/fns are not convertible and are a compile error as a
controller argument — `CompileError::NonDataControllerArg`). `HostValue` mirrors
`Value::Record` from RFC-0027, so returned data drops straight into the view.

### 2. The `Controller` trait and the generated shim

```rust
// byard-core::bridge
pub trait Controller: Send + Sync {
    /// Stable type name used as the `inject` key (the struct's ident).
    fn type_name(&self) -> &'static str;
    /// Dispatch one async method by name. Returns a boxed future that yields
    /// either a success or an error HostValue. Never blocks the caller.
    fn invoke(
        &self,
        method: &str,
        args: Vec<HostValue>,
    ) -> BoxFuture<'static, Result<HostValue, HostValue>>;
}
```

`#[byard_controller]` generates `impl Controller for WeatherApi`:
- `type_name` → the struct ident.
- `invoke` → a `match method { "forecast" => …, _ => Err(unknown-method) }` that
  converts `args` into the method's Rust parameter types (via a generated
  `FromHostValue`), `Arc::clone`s `self` into the future, `.await`s the method,
  and maps `Ok(T)`/`Err(E)` through `Into<HostValue>`. `T` and `E` implement
  `IntoHostValue` (derivable; RFC-0029 forecast types will).
- The existing `BYARD_FIELDS`/`byard_field_type` metadata is **retained**.

The macro crate still has **no dependency on `byard-core`/`byard-compiler`**: the
generated code references `::byard::bridge::{Controller, HostValue, …}` from the
*app* crate's dependency graph (app → byard → byard-core is a forward edge,
INV-1 preserved). A static-assertion test in `byard-macro/tests` confirms the
emitted `impl` compiles against the façade.

### 3. Registry and `inject` wiring

`byard-core` gains a `ControllerRegistry` (`HashMap<&'static str, Arc<dyn
Controller>>`), owned by the engine and reachable from the logic thread. `App::
provide(c)` inserts `c.type_name() → Arc::new(c)`. At view mount, the interpreter
seeds the root `Env` with one ambient `Value::Controller(ControllerId)` per
registered controller, keyed by type name — so `resolve_inject` (unchanged)
finds it and `inject WeatherApi as api` binds `api` to the handle. `Value` gains
one variant, `Controller(ControllerId)`; `ControllerId` is a `u32` index into the
registry (arena-friendly, `Copy`, thread-local read on the logic thread only —
consistent with INV-2, the handle is never dereferenced off the logic thread; it
only *schedules* work onto the pool).

### 4. Grammar: the async-call expression (RFC-0002/0003 extension)

A new statement form, valid only in **action position** (an event body or a
result arm — never in a pure value/memo, because it is an effect):

```
async_call := postfix "(" args ")" result_arms?
result_arms := ( "ok" IDENT block )? ( "err" IDENT block )?
```

Lowered to `Expr::ControllerCall { handle, method, args, ok_arm, err_arm, span }`.
Arms reuse RFC-0019's callback-prop binding (`Value::Fn` bodies with one bound
param). Omitting both arms is legal (fire-and-forget). Placing a
`ControllerCall` in a pure/memo context is `CompileError::EffectInPureContext`
(mirrors RFC-0027's `EffectInPureLambda`).

### 4b. Lifecycle effects: `on mount` / `on unmount` (new)

A controller call needs an **entry point** at view start (load data when the
screen appears). `byld` has no lifecycle primitive today (no `on mount` exists in
`interp/`), so this RFC adds two structural effects (RFC-0018) valid at
View-member position:

```
lifecycle := ("on" ("mount" | "unmount")) "=>" action
```

Lowered to `Member::Lifecycle { on_mount: bool, action }`. `on mount` runs its
action once when the enclosing structural scope mounts (including remount under
`when`/`for`); `on unmount` runs when it unmounts. Both are **action position**
(effects), so they may contain a `ControllerCall`. This is the trigger the
RFC-0029 examples use to fetch on appearance. `on mount`/`on unmount` compose
with timers (RFC-0029 O4): a timer *is* a repeating mount-scoped effect.

### 5. Dispatch — the missing plumbing (the core of this RFC)

Sequence, all reference-free:

1. **Action runs on the logic thread.** Evaluating `ControllerCall` converts args
   `Value → HostValue`, allocates a `ContinuationId`, stores the `(ok_arm,
   err_arm)` closures in the interpreter's continuation table keyed by that id,
   and calls `ctx.spawn_controller(controller_id, method, host_args,
   continuation_id)`.
2. **Spawn on the Tokio pool.** `spawn_controller` looks up `Arc<dyn Controller>`
   in the registry, then `relay.io_handle().spawn(async move { let r =
   c.invoke(method, args).await; tx.send(Box::new(ControllerReply {
   continuation_id, result: r }) as IoResult) })`. The blocking/async work never
   touches the logic or render thread (INV-12). `tx` is `relay.io_result_sender()`.
3. **Drain on the logic thread — new wiring.** `relay.rs::spawn_logic_from_view`
   is amended: at the **top of each tick**, before `evaluate_tick`, it drains
   `relay.try_recv_io_result()` and forwards `ControllerReply`s to
   `runtime.apply_io_results(...)`. (Non-`ControllerReply` results — e.g.
   `DecodedImage` — are left for the render-thread drain, matched by downcast, so
   the two consumers do not fight; see §7.) This closes the exact gap the audit
   found: today this drain does not exist on the logic side.
4. **Apply on the logic thread.** `Interpreter::apply_io_results` (already
   present, `eval.rs:995`) downcasts each `ControllerReply`, looks up the
   continuation by id, converts `HostValue → Value`, and runs the `ok`/`err` arm
   as an action — writing `var`s via `write_var`, marking them dirty (RFC-0004
   tick step 3). The next pull re-renders. The continuation is then removed
   (one-shot).

All comparison/writing happens on the logic thread reading `Signal`s (INV-2). The
result payload crossing `mpsc` is `Send` `HostValue` (INV-2).

### 6. Tick ordering (RFC-0004 consistency boundary)

I/O results are applied at **tick step 0** (before input processing and before
the pull), so a result and a same-tick input both mark before the single pull,
preserving D1's "tick = consistency boundary" and E6's intra-tick step ordering.
A result that arrives mid-frame simply lands on the next tick — never mid-pull.

### 7. Coexistence with the render-thread image drain (INV-12)

The `IoResult` channel is shared. To avoid the logic thread and render thread
racing for the same message, results are **typed by consumer**: `ControllerReply`
is drained by the logic thread; `DecodedImage` by the render thread. Each drain
downcasts and **re-queues** (or ignores) messages not addressed to it. Chosen
mechanism: split into two channels at the `Relay` level — `io_result_tx` (logic:
controller replies) and `decode_result_tx` (render: image decodes) — so neither
side sees the other's traffic. This is a small, backward-compatible `Relay`
refactor (the image path moves to `decode_result_*`; existing tests updated).

### 8. Dev vs Prod (RFC-0001 §7.3)

- **Dev (interpreted):** exactly the above — the registry + type-erased dispatch,
  because the interpreter cannot statically know app types.
- **Prod (transpiled, Phase 4):** `api.forecast(x) ok r => {…}` transpiles to a
  direct `io_handle.spawn` of `api.forecast(x)` with the arm as the completion
  closure and a `Signal` write — no registry, no `HostValue` erasure. The surface
  and the semantics are identical; only the dispatch is monomorphized. The
  `HostValue` boundary is a dev-mode implementation detail, not a language
  concept, so the transpiler is unaffected.

### 9. Diagnostics (INV-5, in `byard-compiler`)

`UnresolvedInject` (existing) for a missing controller; `UnknownControllerMethod`
(Levenshtein, per D4); `NonDataControllerArg` (passing a signal/fn/view);
`EffectInPureContext` (call in a memo); `ControllerArmTypeMismatch` (arm param
used incompatibly). Unknown method at **runtime** (dev registry mismatch) yields
an `err` arm with a structured error record, never a panic (INV-4).

## Drawbacks

- **Type-erased dev dispatch.** Args/results round-trip through `HostValue`,
  costing conversions and losing static typing at the boundary in dev. Mitigated:
  the conversions are shallow, dev is not the perf target (RFC-0001 §7.3), and
  prod transpilation removes them entirely.
- **A shared-channel refactor.** Splitting the `IoResult` channel touches
  `relay.rs` and the image path. Small and well-tested, but it is core-thread
  code — must keep every existing relay invariant/test green.
- **Registration boilerplate.** The app now needs a real `main.rs`. This is
  inherent to the two-file model; the scaffold change makes it one line per
  controller.
- **Effect-only calls.** Controllers cannot be called from pure/memo context, so
  "derive a value by calling Rust" is not expressible — by design (purity). Data
  must flow call → `var` → view.

## Rationale and alternatives

**Why a `HostValue` wire type instead of passing `Value` across?** `Value` is
`!Send` (it holds `Signal`s, RFC-0001 §5.1) and lives in `byard-compiler`; a core
trait using it would violate both INV-2 (Send across threads) and INV-1 (no core→
compiler dependency). A neutral `Send` data enum in `byard-core` is the minimal
type that both the app's generated shim and the interpreter can speak. It also
gives the transpiler a clean seam.

**Why a runtime registry rather than compile-time binding in dev?** The dev
interpreter is generic over any `.byd` and cannot know an app's Rust types at
compile time (that is precisely why prod transpiles). A registry keyed by type
name is the same React-Context provider model `inject` already uses — this RFC
generalizes providers from values to handles rather than inventing a new concept.

**Why result arms (`ok`/`err`) instead of returning a `Promise`/awaitable?**
Reference-freedom (RFC-0003) forbids handles; an awaitable is a handle. Arms are
callback props (RFC-0019) — already reference-free and typed — and they make the
success/error split explicit at the call site, which is exactly what a network
call needs.

**Why drain I/O results at tick step 0?** D1 makes the tick the consistency
boundary; applying results before the pull (with inputs) means a frame reflects
all marks coherently. Applying mid-pull would break idempotent-mark ordering
(E6).

**Alternative rejected: reuse the single shared `IoResult` channel with
re-queueing.** Re-queueing risks livelock and ordering surprises between two
threads. Two typed channels are simpler and keep each thread's drain O(its own
traffic).

## Prior art

- **RFC-0001 §5.1** — the origin: Tokio pool + `tokio::sync::mpsc` results. This
  RFC implements it.
- **Flutter platform channels / method channels:** a `HostValue`-like erased
  codec crossing a language boundary with async replies. Direct model for §1–§2,
  minus the serialization (Byard shares memory).
- **React Server Actions / SolidJS `createResource`:** the "call → pending →
  resolved into reactive state" flow the result arms formalize.
- **Tauri commands / `wasm-bindgen` async:** Rust async exposed to a declarative
  front end via generated glue — the macro-shim approach.
- **Elm effects / `Cmd`:** effects are values scheduled by the runtime, results
  delivered as messages that update the model — the ok/err arm model.

## Resolved questions

- **Before merge:**
  - [x] **Boundary type.** **A `Send` `HostValue` enum in `byard-core`**, not
    `Value` and not serde bytes. Preserves INV-1/INV-2 and shares memory.
  - [x] **Where results are applied.** **Logic thread, tick step 0**, via the
    now-wired `try_recv_io_result → apply_io_results` drain. Fixes the audited
    gap.
  - [x] **Call ergonomics.** **`ok name => {} err e => {}` arms** (RFC-0019
    callbacks), both optional; fire-and-forget allowed.
  - [x] **Purity.** Controller calls are **effects**, legal only in action
    position; `EffectInPureContext` otherwise.
  - [x] **Channel sharing with image decode.** **Split into two typed channels**
    (`io_result` logic / `decode_result` render). No re-queueing.
  - [x] **Macro dependency direction.** Generated `impl` references the façade
    (`::byard::bridge::*`); `byard-macro` gains no dep on core/compiler (INV-1).

- **During implementation:**
  - [x] **Continuation lifetime.** One-shot: stored on call, removed on apply. A
    view unmount (hot-reload structure change, RFC-0003 E5) drops pending
    continuations; a late reply for a dropped id is discarded with a diagnostic
    (INV-4), never applied to a stale `var`.
  - [x] **Arg/return derive.** `IntoHostValue`/`FromHostValue` derives for
    controller method params/returns; scalars, `String`, `Vec<T>`, and
    `#[derive(HostValue)]` structs supported. Unsupported types are a **compile
    error in the app crate** (clear, at the controller definition site).
  - [x] **Cancellation.** v1: no explicit cancel; a superseded call's continuation
    is dropped on unmount and its reply ignored. Explicit cancellation is a future
    possibility.
  - [x] **Back-pressure.** The Tokio pool is bounded by its worker count; a flood
    of calls queues on the pool, not the logic thread. A per-controller in-flight
    cap is deferred (Future possibilities).

## Future possibilities

- **Streaming/subscription methods** (`api.subscribe(...)` yielding many
  `HostValue`s) for websockets/SSE — a `stream` arm delivering repeatedly until
  unmount.
- **Explicit cancellation tokens** bound to view lifetime.
- **`provide` scoping** below the root (per-subtree controllers) reusing the `Env`
  chain.
- **Prod transpiler lowering** (Phase 4) that monomorphizes the dispatch and
  deletes the `HostValue` round-trip.
- **Typed controller metadata → LSP** so `api.` autocompletes methods and
  argument types across the boundary (building on the retained `BYARD_FIELDS`).
